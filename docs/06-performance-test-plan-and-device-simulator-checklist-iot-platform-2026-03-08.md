## 1. 文档目标

本文档用于指导物联网管理系统上线前的性能验证，包含：

- 压测目标
- 压测场景
- 指标口径
- 压测实施步骤
- 设备模拟器脚本清单
- 压测结果输出模板

---

## 2. 压测目标

基于当前方案，压测目标如下：

- 注册设备规模：`200,000`
- 峰值在线设备：`60,000`
- 遥测接入峰值：`10,000 msg/s`
- 命令下发峰值：`500 cmd/s`
- 查询链路目标：设备列表查询 `P95 < 500ms`
- 命令链路目标：下发受理到投递 `P95 < 2s`

---

## 3. 压测范围

## 3.1 连接类压测

- MQTT 建连压测
- MQTT 重连风暴压测
- 长连接稳定性压测
- 鉴权接口压测

## 3.2 消息类压测

- 单设备周期上报压测
- 多设备并发遥测压测
- 事件风暴压测
- 影子同步压测

## 3.3 控制类压测

- 单设备命令下发压测
- 批量命令下发压测
- 命令 ACK/结果回执压测
- 命令超时与失败补偿验证

## 3.4 查询类压测

- 设备列表分页查询
- 设备详情查询
- 命令历史查询
- 告警列表查询
- OTA 任务列表查询

## 3.5 稳定性与故障场景压测

- Broker 单节点故障下长连接恢复
- Kafka Broker 单节点故障下消息积压恢复
- Redis 主从切换影响验证
- PostgreSQL 主备切换影响验证
- 服务滚动升级期间链路稳定性验证

## 4. 核心指标口径

| 指标 | 定义 |
|---|---|
| 连接成功率 | 成功建立 MQTT 连接数 / 建连总尝试数 |
| 鉴权成功率 | 成功通过鉴权数 / 鉴权请求总数 |
| 遥测接入成功率 | 成功进入 Kafka 的消息数 / 上报总消息数 |
| Kafka lag | 消费滞后量 |
| 命令受理时延 | API 请求进入命令服务到返回受理结果 |
| 命令投递时延 | 命令受理到 MQTT Broker 成功投递 |
| 查询 P95 | 后台接口 95 分位响应时间 |
| 服务错误率 | 5xx/业务错误比例 |

## 5. 压测场景设计

## 5.1 场景 A：基础连接压测

### 目标

验证系统可稳定支撑 `60,000` 设备在线。

### 操作步骤

- 分批拉起模拟设备连接
- 每批 `5,000` 连接递增
- 持续观察连接数、鉴权成功率、Broker CPU/内存
- 稳定运行 `24` 小时

### 验收条件

- 连接成功率 >= `99.5%`
- Broker 无明显抖动
- 重连率可控

## 5.2 场景 B：遥测吞吐压测

### 目标

验证系统在 `10,000 msg/s` 下持续运行能力。

### 操作步骤

- 每个模拟设备按预设频率上报遥测
- 逐步增加消息速率到 `10,000 msg/s`
- 观察 Kafka 入站速率、Broker 压力、消费者 lag
- 持续运行 `30~60` 分钟

### 验收条件

- Kafka lag 不持续增长
- 服务错误率在阈值内
- 时序写入正常

## 5.3 场景 C：命令链路压测

### 目标

验证 `500 cmd/s` 下发链路时延与稳定性。

### 操作步骤

- 通过后台/API 批量创建命令
- 模拟设备回执 ACK 与执行结果
- 观察命令状态流转完整性与时延

### 验收条件

- 命令投递 `P95 < 2s`
- 状态机无异常跳变
- 无明显幂等重复问题

## 5.4 场景 D：查询压测

### 目标

验证管理后台主要接口性能。

### 操作步骤

- 使用压测工具执行设备分页、详情、命令记录查询
- 模拟多租户、多筛选条件请求
- 观察 API Gateway、query-service、PG、Redis 指标

### 验收条件

- 设备列表查询 `P95 < 500ms`
- 设备详情查询 `P95 < 300ms`
- 系统无明显慢查询爆发

## 5.5 场景 E：故障演练压测

### 目标

验证核心组件故障时系统可恢复性。

### 操作步骤

- 在稳定压测下摘除一个 Broker 节点
- 在稳定压测下摘除一个 Kafka Broker
- 在稳定压测下重启一个核心 Go 服务
- 观察恢复时间、连接数波动、lag 变化

### 验收条件

- 主链路不中断或快速恢复
- 无大面积消息丢失
- 无大面积命令失败

## 6. 设备模拟器脚本清单

建议建设独立设备模拟器工程，至少包含如下脚本：

| 脚本名称 | 用途 |
|---|---|
| `sim-connect` | 批量创建设备连接 |
| `sim-heartbeat` | 周期发送心跳 |
| `sim-telemetry` | 周期发送遥测 |
| `sim-event-storm` | 短时间发送事件风暴 |
| `sim-shadow-report` | 上报 Reported 状态 |
| `sim-command-ack` | 模拟命令 ACK 与执行结果 |
| `sim-ota-client` | 模拟 OTA 通知、下载、回执 |
| `sim-reconnect-storm` | 模拟断线重连风暴 |
| `sim-chaos-client` | 模拟弱网、乱序、重复消息 |

---

## 7. 脚本参数建议

## 7.1 通用参数

- `--broker`
- `--tenant`
- `--product`
- `--devices`
- `--qps`
- `--payload-size`
- `--duration`
- `--qos`
- `--reconnect-rate`
- `--report-interval`

## 7.2 脚本示例

```bash
sim-connect \
  --broker tcp://mqtt.example.com:1883 \
  --tenant t001 \
  --product p001 \
  --devices 5000 \
  --duration 30m
```

```bash
sim-telemetry \
  --broker tcp://mqtt.example.com:1883 \
  --tenant t001 \
  --product p001 \
  --devices 10000 \
  --qps 2000 \
  --payload-size 512 \
  --duration 60m
```

## 8. 压测执行步骤

### 第一步：准备环境

- 清理无关流量
- 固定中间件版本与服务版本
- 确认监控与日志正常
- 准备压测机与网络带宽

### 第二步：基线测试

- 先做小规模基线压测
- 记录 CPU、内存、带宽、lag、时延
- 校正压测脚本与监控口径

### 第三步：逐级放量

- 按 `10% -> 30% -> 60% -> 100%` 逐级放量
- 每一档至少稳定运行 `15~30` 分钟
- 达到瓶颈后记录原因与阈值

### 第四步：故障演练

- 在稳定负载下执行中间件与服务故障演练
- 记录恢复时间与影响范围

### 第五步：结果复盘

- 输出瓶颈点
- 输出参数调整建议
- 更新资源规划表

## 10. 风险与注意事项

- 压测前必须冻结版本，避免结果失真
- 模拟器要尽可能贴近真实设备行为
- 重连风暴是 IoT 平台必须重点验证的场景
- 命令链路要单独测，不要混在纯上报压测里
- 故障演练必须在受控环境进行
